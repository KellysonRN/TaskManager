# 04.2 — Authentication: Expose Endpoint + Data Seed (Backend)

All unit tests are green. Now expose the login endpoint and seed an initial user.

## Your task

1. Create `POST /api/auth/login` endpoint.
2. Protect existing task endpoints with `[Authorize]`.
3. Create a database seed with one initial user.
4. Add integration tests for the auth endpoint.

Do **not** break existing tests.

---

## Part A — Auth Endpoint

### Controller

Create `AuthController` in `TaskManager.Api/Controllers/`:

```
POST /api/auth/login
  Body: { "email": string, "password": string }
  200 OK  → { "token": string }
  400 Bad Request → { "error": string }   (validation error)
  401 Unauthorized → { "error": string }  (wrong credentials)
```

- Controller must depend on `LoginHandler` (or its interface).
- `ExceptionHandlingMiddleware` must map `UnauthorizedException` → `401 Unauthorized`.
- `ExceptionHandlingMiddleware` must map `ValidationException` → `400 Bad Request`.

### Protect task endpoints

Add `[Authorize]` to `TasksController`.
Configure JWT Bearer authentication in `Program.cs` using the existing `JwtSettings`.

---

## Part B — Data Seed

Create `DataSeeder.cs` in `TaskManager.Infrastructure/Persistence/`:

- Seed exactly **one** user on application startup (only if the users table is empty).
- Initial credentials:
  - Email: `admin@taskmanager.dev`
  - Password: `Admin@123`
- Password must be stored **hashed** using `IPasswordHasher` (BCrypt).
- Call the seeder from `Program.cs` after `app.UseAuthentication()`.

Also create the `UserEntity` EF Core configuration and add `DbSet<UserEntity> Users` to `TaskManagerDbContext`.

---

## Part C — Integration Tests

Add to `TaskManager.IntegrationTests/`:

| # | Scenario | Expected status |
|---|---|---|
| 1 | `POST /api/auth/login` with valid credentials | `200 OK` + non-empty `token` |
| 2 | `POST /api/auth/login` with wrong password | `401 Unauthorized` |
| 3 | `POST /api/auth/login` with unknown email | `401 Unauthorized` |
| 4 | `POST /api/auth/login` with empty email | `400 Bad Request` |
| 5 | `GET /api/tasks` without token | `401 Unauthorized` |
| 6 | `GET /api/tasks` with valid Bearer token | `200 OK` |

Use `WebApplicationFactory<Program>` with the same in-memory SQLite setup as existing integration tests.
For authenticated requests, obtain a real token from the login endpoint in the test setup.

---

## Architecture constraints

- `AuthController` must be thin — no business logic.
- Seeder must not run during integration tests — detect via environment or configuration flag.
- For each change, explain what was done and why.
